The week view is a mouse-to-time translator with a draft layer in the middle.

If you remember only one thing, remember this: the grid does not store pixels. It stores event dates. Pixels from the mouse become dates, dates become a temporary draft, and that draft either opens the form or saves through the event pipeline.

Mental Model

A beginner can read the week view as four stacked systems. They run together in the UI, but each has a different job.

1. The visible grid

`WeekView` now keeps the outer week setup separate from `WeekGridInteraction`. The interaction boundary owns date calculations, draft providers, shortcuts, sidebar wiring, and the grid.

WeekView.tsx

2. The date calculator

`useDateCalcs` turns a mouse X/Y position into a day and time. X picks the column. Y picks the minute. The minute is rounded down to the grid step.

useDateCalcs.ts

3. The draft layer

A draft is the temporary version of an event while you are clicking, dragging, resizing, or editing. It lets the UI move first, then decide later whether to save.

useDraftState.ts

4. The save pipeline

Saves go through Redux actions, sagas, repository selection, then local storage or the remote API. The event is marked pending so it cannot be dragged again while the save is in flight.

event.sagas.ts

Coordinate Math

The week view constantly translates between two worlds: screen coordinates and calendar dates. This is the "mouse through the grid" part.

X axis: which day column? Y axis: which minute?
A mouse point becomes a calendar moment. The draft block is then rendered from its `startDate` and `endDate`, not from the original mouse point.
Day column x - gridLeft - GRID_MARGIN_LEFT -> dayIndex

`useDateCalcs.getDateByX` subtracts the measured left edge and the time-column margin, then walks the measured column widths to find the day.

Minute of day ((y - gridTop + scrollTop) / hourHeight) * 60

Gesture Flows

Pick a gesture. Each panel walks from the user's hand motion to the code path that handles it.

Mouse down on empty timed grid

`MainGrid` ignores right-clicks and existing drafts. Otherwise it starts a timed draft from the pointer position. MainGrid.tsx:49

Pointer becomes a start time

`dateCalcs.getDateByXY` chooses the day and snapped minute. The initial end time is 15 minutes later. MainGrid.tsx:62

A default event is assembled

`assembleDefaultEvent` creates a timed grid event with default priority, origin, user, and grid position metadata. event.util.ts:71

Creation is treated like resizing

The code dispatches `draftSlice.actions.startResizing` with `dateToChange: "endDate"`. Dragging downward while creating an event is literally resizing the temporary event's end. draft.slice.ts:36

Mouse up opens the form

For a new timed draft, mouse-up opens the event form instead of saving immediately. The form owns title, priority, recurrence, and final save. useGridMouseUp.ts:95

Click or drag is decided first

`useGridEventMouseDown` waits for either a 750 ms hold or more than 25 px of movement. A quick release opens the form. Hold or move starts a drag. useGridEventMouseDown.ts:31

The real event becomes a draft

`MainGridEvents.handleDrag` stores the original mouse offset and dispatches `startDragging`. Pending events are blocked here. MainGridEvents.tsx

Document-level mousemove takes over

`Draft` mounts `useGridMouseMove`, which listens for mouse movement while drafting and calls `actions.drag`. useGridMouseMove.ts:14

Dates are rewritten, not pixels

`useDraftActions.drag` maps the mouse to a new start time, preserves the old duration, prevents timed events from overflowing past midnight, and updates the draft only when the snapped start or end actually changed. useDraftActions.ts

Mouse up saves if it really moved

`useGridMouseUp` checks `dragStatus.hasMoved`. Same-instant timed movement does not set that marker, moved timed events submit, and clicked existing events open the form. useGridMouseUp.ts:18

Invisible handles catch the mouse

Timed events have hidden top and bottom strips. Top means `startDate`; bottom means `endDate`. GridEvent.tsx:210

The draft remembers which edge is moving

`dateBeingChanged` is set to `startDate` or `endDate`, then `isResizing` starts. GridDraft.tsx:82

Resize rejects impossible movement

Timed resize ignores no-op movement, same-start movement, and cross-day movement. All-day resize is more permissive because it moves by days. useDraftActions.ts:457

The active edge can flip

If you drag the start past the end, or the end before the start, the code flips which date it is changing so the event stays valid. useDraftActions.ts:489

Mouse up follows the same submit rules

Resized existing events submit on release. New drafts still open the form. useGridMouseUp.ts:87

The form reads the event dates

`EventForm` calls `getFormDates` when the event changes. That fills local picker state for start time, end time, and selected dates. EventForm.tsx:101

Time pickers update the event draft

`TimePickers` calls `mapToBackend` after a time selection, then writes new `startDate` and `endDate` into the draft. TimePickers.tsx:79

Bad time order is auto-corrected

If changing start would collide with end, or changing end would collide with start, the companion time is adjusted to preserve a usable duration. web.datetime.util.ts:43

Submit maps picker state back to stored dates

On save, `EventForm.onSubmitForm` calls `mapToBackend`. All-day one-day events use an exclusive next-day end date. web.date.util.ts:180

Save Pipeline

Once a draft is ready to save, the path is intentionally centralized. Components do not choose storage directly.

GridDraft / EventForm
Submit the draft event from the form or mouse-up handler.
useDraftConfirmation
For recurring events, ask whether the change applies to this event or more of the series.
useDraftActions.submit
Decide create, update, discard, or reopen form.
event slice
Dispatch `createEventSlice.request` or `editEventSlice.request`.
event saga
Optimistically update visible state, mark the event pending, call the repository, then clear pending.
repository
Choose local IndexedDB or remote API based on auth, Google revocation, and backend availability.
Decision Code Beginner translation
Is this new? !draft._id -> CREATE No saved ID means Compass needs to create a new event.
Is it pending? pendingEventIds.includes(draft._id) If a previous save is still happening, throw away the new movement instead of stacking writes.
Did it change? DirtyParser.isEventDirty If the draft is identical to the original event, do not save.
Where does it save? getEventRepository(sessionExists) Local storage for local/offline modes, remote API for authenticated users.

Where State Lives

A lot of confusion comes from the word "draft" meaning a few nearby things. This table separates them.

Thing What it means Where to look
Event entities Normalized saved events used by the week selectors. The week slice stores IDs, and the entity map stores event objects. eventsEntitiesSlice, getWeekEventsSlice
Redux draft status The current draft event plus activity: grid click, dragging, resizing, keyboard edit, and so on. draft.slice.ts
DraftProvider local state Live gesture state: is dragging, is resizing, which date edge is moving, whether the form is open. useDraftState.ts
Grid event position Render-only metadata used to calculate overlap, width, drag offset, and visual placement. web.event.types.ts, position.util.ts
Pending event IDs IDs currently being saved. Pending events show wait behavior and cannot be dragged/resized again. pending.slice.ts
EventForm picker state Local form state for selected dates and times. It is mapped back into event dates on save. EventForm.tsx, TimePickers.tsx
Small gotcha: the repo also has an Elf `eventsStore` used by some shared floating-form and day-view paths. The main week grid path described here uses Redux week IDs, Redux event entities, Redux draft status, and local DraftProvider state.

Smoothness Notes

This section is now a status snapshot for this branch. Some earlier pressure points have been fixed, and a few are still worth knowing before the next round of performance work.

Mousemove still drives draft state, but no-op timed drags are guarded
During drag and resize, document-level mousemove calls `useDraftActions.drag` or `resize`, and those can update React draft state. This branch adds guards so timed drags that snap to the same start/end, including the same instant represented with a different offset, do not call `setDraft` or mark the drag as moved again.
useGridMouseMove.ts:14
Grid measurement objects are stable when the size is stable
`useGridLayout` still measures the all-day row, all-day columns, and main grid, but it now avoids state replacement when a ResizeObserver reports the same rectangle. It also memoizes `colWidths`, `gridRefs`, and `measurements`, which helps memoized grid consumers keep their boundary.
useGridLayout.ts
Timed overlap handling is now selector work
Timed events are still copied, sorted, grouped by overlap, and assigned horizontal lanes. The important change is where it happens: `selectGridEvents` now returns overlap-adjusted events, so `MainGridEvents` does not redo that work during render when the selected week events have not changed.
event.selectors.ts
Smart scroll no longer stores pointer position in React state
Timed drag auto-scroll now stores only the mouse Y value in a ref and schedules its scroll loop with `requestAnimationFrame`. Mousemove no longer causes a React re-render just to remember pointer position. Week-edge navigation still has its own mouse tracking, timer, and progress state, so that remains a good future cleanup target.
useDragEventSmartScroll.ts
Performance is tracked commit by commit
This branch adds `bun run perf:week`, which seeds a browser run and records empty load, heavy load, create, grid-create, drag, and resize scenarios. Use `--compare latest` after each focused commit to keep the before/after trail. The latest local rerun for this branch is `2026-05-13T21-05-40-575Z-simplify-week-performance-cleanup-rerun.json`.
performance/week-view/README.md
Focused tests now cover the new performance boundaries
This branch added focused checks for stable grid layout objects, memoized overlap-adjusted selectors, no-op timed drag updates, and smart-scroll pointer tracking without re-rendering. Remaining gaps are full-flow week drag/release behavior, edge navigation, and resize no-op guarding.

Beginner Dictionary

The short version of the terms you will see while reading the source.

Grid event
An event that belongs on the calendar grid. It has `startDate`, `endDate`, and extra web-only `position` data so the UI can place it.
Draft event
A temporary event used while the user is creating, dragging, resizing, or editing. The draft may be saved, discarded, or used only to open the form.
Pending event
An event currently waiting for storage or API confirmation. Compass blocks new drag/resize gestures on pending events to avoid conflicting writes.
Exclusive all-day end date
A one-day all-day event is stored as start date plus next-day end date. Example: May 13 all day is stored as May 13 to May 14.
Optimistic update
The UI updates immediately before the save finishes. If the save fails, the saga rolls the visible event back or removes the temporary event.
