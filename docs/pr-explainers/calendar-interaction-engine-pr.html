PR 1776 - Week calendar interaction engine cutover

A field guide to the new Week interaction engine

This PR changes who is in charge when you drag or resize saved events in Week view. Before, the old draft system did live movement work while the pointer was moving. After this PR, a new interaction engine owns the live pointer session, moves a visual clone immediately, and only asks the rest of the app to save or open a form after the interaction ends.

The shortest useful explanation

If someone joins this PR cold, this is the mental model to give them before they read a single file.

What changed?

Saved Week events now have a dedicated interaction path. When the user presses on a saved timed or all-day event, the new Week interaction adapter decides whether it owns that gesture. If it does, it blocks the older saved-event drag/resize handlers from also responding. The shared engine then tracks the pointer, waits to see if the gesture is a click or real movement, mounts a clone overlay when motion starts, updates that overlay on animation frames, and returns a final result on pointer release.

The rest of the app still owns the familiar calendar behavior that was not part of this migration: empty-grid event creation, draft-event editing, keyboard editing, form UI, and Someday/sidebar drag and drop. This is not a whole calendar rewrite. It is a precise handoff for saved event drag and resize in Week view.

Why do it?

The goal was to keep drag and resize motion snappy. During sustained pointer movement, the engine should not be asking React to re-render, asking Redux to dispatch, sending save requests, or mutating random DOM outside the allowed overlay/source-placeholder work. React can still render at normal boundaries, and saves can still happen at the end. The new rule is: while the pointer is moving, the engine owns the motion.

The core idea The engine is not the calendar. It is the motion machine. Week provides the calendar knowledge: columns, dates, snapping, smart scroll, edge navigation, recurrence/form behavior, and how to turn a final visual position back into an event.

What the plan asked for

The implementation plan was docs/superpowers/plans/2026-05-18-calendar-interaction-engine.md. It asked for a clean rebuild of a Week prototype as a reusable engine, not a wholesale branch move.

Build a shared engine

Put generic pointer-session logic in packages/web/src/common/calendar-interaction/. This layer may know about pending sessions, motion sessions, animation frames, overlay lifecycle, metrics, and cancellation. It must not know Week-specific date math or layout details.

Build a Week adapter

Put Week-specific code in packages/web/src/views/Week/interaction/. This layer owns saved timed drag, timed resize, all-day drag, and all-day resize. It knows how Week columns, snap intervals, event markup, recurrence, form reopening, all-day exclusive end dates, and edge navigation work.

Cut over without duplicate owners

Once the new adapter owns a saved-event interaction, remove or disable the older live-motion path for that same saved-event behavior. The old system should remain only for flows still outside this migration.

Measure the hard parts

Sustained motion has strict gates: zero React commits, zero Redux dispatches, zero save requests, zero unexpected DOM mutations, zero long tasks, and animation-frame work under tight budgets. Commit and week-navigation render are measured separately because those phases are allowed to do app work.

The architecture in layers

The PR works because responsibilities are split very sharply. The shared engine is deliberately small in calendar knowledge. Week is deliberately rich in calendar knowledge.

1. Week event components

Render saved timed and all-day event boxes. If an event has a real ID and is not pending, draft, or placeholder, it gets registration attributes and a registry ref.

2. Week boundary

Wraps the grid and listens in capture phase for pointer events. It asks the adapter if the pointer should be owned and blocks older handlers only when ownership is accepted.

3. Week adapter

Resolves the target, builds Week geometry, creates Week visual state, runs smart scroll and edge navigation, and converts final visuals back into event dates.

4. Shared engine

Runs the generic session state machine: pending, click, activation, animation-frame updates, commit, cancel, overlay teardown, and metrics.

Ownership matrix

This is the PR's most important review concept. A gesture should have one live-motion owner, not two.

Saved timed drag new engine

Drag an existing timed event vertically or across day columns.

Saved timed resize new engine

Resize existing timed event from top or bottom handle.

Saved all-day drag new engine

Move existing all-day event horizontally by visible day columns.

Saved all-day resize new engine

Resize existing all-day event from left or right handle.

Pending events existing path

Not registered as engine targets; pending UI remains protected.

Draft events existing path

Draft drag and resize stay in the old draft editing flow.

Empty grid existing path

Click or drag empty time slots still creates drafts outside the engine.

Someday/sidebar existing path

Sidebar drag and drop was left for a future adapter.

Timed drag

Timed drag is the main "move this meeting" flow. The adapter starts from the event's original start and end time. As the pointer moves, it converts vertical distance into snapped minutes and horizontal distance into the nearest rendered day column. Duration is preserved. If the pointer gets near the top or bottom of the timed grid, smart scroll adjusts the scroll container and feeds that scroll delta into saved time math without making the overlay drift away from the pointer.

RuleBehavior
Snap15-minute slots.
DurationOriginal duration is preserved.
Day choiceNearest rendered day column by horizontal center.
ClampStart time cannot move outside the day.

Edge navigationAfter 500ms dwell at left or right edge, request previous or next week.

Timed resize

Timed resize changes the start or end time without moving the event to another day. The user grabs an invisible top or bottom handle. The overlay writes a new height and, for top-edge resize, a new vertical transform. If the dragged edge crosses over the opposite edge, the active edge flips so resizing remains continuous instead of getting stuck.

RuleBehavior
HandlesTop changes start; bottom changes end.
MinimumOne snap slot.
Edge flipCrossing through the opposite edge flips start/end ownership.
CommitOnly startDate and endDate time are changed.
OverlayHeight and time label update immediately.

All-day drag

All-day drag is horizontal. It uses the visible all-day row columns and moves by whole day positions. The event's span is preserved, meaning a two-day all-day event remains two days long after it moves. If the user drags near the week edge and dwells there, the adapter requests week navigation and keeps the active overlay session alive.

All-day resize

All-day resize works in inclusive visual day indexes while the user is moving, then commits back to Compass's date-only all-day event format with an exclusive end date. It supports left and right handles, one-day minimum, edge flip, clipped/out-of-week spans, and legacy same-day end date normalization only after visible resize movement.

Commit routing: what happens after pointer release?

1. No movement

If the visual result did not really move, do not save. Open the existing event just like a normal click.

2. Form was already open

If the event moved and the form was open before the interaction, reopen the changed event as a draft instead of saving immediately.

3. Recurring event

If it moved, no form was previously open, and the event is recurring, route through the existing recurrence-scope confirmation.

4. Plain saved event

If it moved and none of the earlier cases apply, submit the edit once after pointer release.

Old path versus new path

Before this PR Saved-event motion could wake the same draft machinery used for editable draft events. That meant live movement could involve React state and Redux-style app work while the pointer was moving.
After this PR Saved-event motion is owned by a native pointer session and a body-level overlay clone. React and Redux rejoin at boundaries: click, cancel, commit, form reopen, recurrence scope, or final save.

Review notes and caveats

These are the things I would want a newcomer to keep in their head while reviewing the PR.

Local checkout caveat While this guide was written, the local branch was one commit ahead of origin/feature/calendar-interaction-engine and had additional unstaged cleanup edits. One untracked helper, weekInteractionMotionState.ts, reads a different global flag name than the committed adapter writes. Treat that as local checkout state, not the clean PR story.
Layout-read metric caveat The performance evaluator can fail if layoutReadsDuringMotion is nonzero, but the original plan warned not to claim layout-read success unless real read sites are instrumented. This is worth reviewing carefully before treating layout reads as fully proven.
Form and recurrence behavior The route is implemented through the boundary controller and existing draft confirmation path: no movement opens the event, moved with prior form open reopens as draft, recurring edits go through scope selection, and plain moved events submit after release. Reviewers should still pay attention to how directly those cases are covered.
What stayed intentionally unchanged Day/Today, Someday, sidebar drag/drop, empty-grid draft creation, draft event drag/resize, backend, sync, auth, MongoDB, Google, and SSE behavior are outside the migration.

The one-page mental model

If you only keep one model, keep this: target registration, pointer ownership, overlay motion, final routing.

  1. Saved events register themselves as engine-eligible DOM targets only when they are real saved events and not pending or drafts.
  2. The Week boundary sees pointer events first. It asks the Week adapter whether the target is a saved timed/all-day drag or resize target.
  3. If the adapter owns the target, the shared engine starts pending. Quick release remains a click. Movement past 25px or a 750ms hold starts motion.
  4. Motion hides the original source event and moves a body-mounted clone with immediate transforms. Week math decides the visual position.
  5. Smart scroll and week edge navigation are handled by the Week adapter during the same active session, not by a second late path.
  6. Pointer release converts visual state into event dates, removes the overlay, restores the source event, and returns one result.
  7. The controller decides whether that result opens the event, reopens a draft form, opens recurrence scope, or saves once.
