What changed

The app stopped rebuilding the calendar while your pointer moves.

The old path treated dragging and resizing as a fast stream of app state changes. Every little pointer move could update a draft in Redux, wake up React, and redraw parts of the Week View. The new path treats pointer motion like an animation: the saved calendar stays still, a temporary overlay moves on top of it, and the real event is saved only after the pointer is released.

React now shows the stable page

React still renders the Week View, the saved events, and the event form. During live drag/resize motion, React is no longer responsible for moving the visible event.

stable shell form preserved

The controller owns the live session

The new WeekInteractionController watches the pointer, keeps tiny visual state, and writes movement to a DOM overlay inside one animation-frame loop.

one RAF loop small visual draft

Redux and saves happen after release

Redux and the backend still own the real saved event, but only once the interaction finishes. The commit adapter preserves existing form, save, and recurring-event behavior.

canonical state later recurrence preserved
The short version The visible motion is now a lightweight DOM overlay. The saved calendar does not churn while your pointer moves. When you let go, the controller converts the visual result into the normal event shape and hands it back to the app.
Beginner glossary

Terms this branch uses

These are the ideas that make the performance work easier to understand before opening the code.

React commit

A moment when React finishes applying UI changes. React commits are normal, but during pointer motion they can steal time from animation frames.

Redux dispatch

A message sent to the global app store. Dispatches are normal for real app changes, but not for every pixel of drag motion.

RAF

Short for requestAnimationFrame. It is the browser's animation clock. V2 does pointer math and DOM writes once per frame.

Visual draft

Tiny temporary information like "this event is 30 minutes lower" or "this all-day block is two columns wider." It is not a full calendar event object.

Overlay

A temporary DOM element that looks like the event being moved. It sits above the saved event and is the only visible thing that moves during live motion.

Canonical event

The real saved event shape, Schema_GridEvent. V2 avoids using this as live-motion state, then creates it at commit time.

The performance move

Before V2, pointer motion went through the whole app.

The core change is ownership. The question is no longer "how do we optimize the old draft path?" The question is "who should own the live pointer session?" V2 answers: the dedicated Week interaction controller.

The saved event is not the moving thing anymore

The saved calendar remains React-rendered. When the controller activates, it hides or placeholders the source element, mounts an overlay, and updates that overlay directly. 

Pointer motion:
  event element registration
  -> controller pending session
  -> overlay mount
  -> RAF visual update
  -> overlay transform/size write
  -> pointerup commit adapter

Architecture map

The new system has four main pieces.

A beginner-friendly way to read this branch is to separate the work into registration, ownership, animation, and commit.

1. Saved events identify themselves

Each event DOM node registers its event id, event data, kind, and resize handles. That gives the controller a way to find the real element without React re-rendering it.

GridEvent.tsx:173 AllDayEvent.tsx:114

2. Boundary captures input first

WeekInteractionBoundary listens in the capture phase, asks the controller whether it owns the pointer, and suppresses the old mouse path while V2 owns motion.

Boundary.tsx:214 Boundary.tsx:287

3. Controller animates the overlay

The controller stores a tiny visual draft, schedules RAF, does math, writes transform or size to the overlay, and records performance counters.

Controller.ts:625 Controller.ts:687

4. Commit adapter resumes app behavior

Pointer-up converts the visual draft into a normal event. The adapter then opens the form, asks for recurrence scope, or submits the save exactly where the app already expects it.

Layer Owns Does not own anymore Why that matters
React Saved calendar UI, event form, normal rendering. Per-frame event movement. React can stay calm while the pointer moves.
Redux/backend The real saved event after commit. Every intermediate drag or resize position. Global state changes once, not dozens of times per second.
V2 controller The live pointer session and visual draft. Long-term event truth. Pointer math stays local and cheap.
DOM overlay The visible moving event during live motion. Saved event data. Motion becomes a direct style write, not app re-rendering.
Interaction life cycle

What happens from click to save

V2 keeps clicks, drags, resizes, recurrence, and form-open behavior separate. That was the key to making performance better without changing the product feel.

Clicking an existing event

On pointer down, the controller starts a pending session. If the pointer comes up without real movement, the controller returns a click result. The boundary then opens the existing event using the normal draft start path. 

Controller.ts:263-271
pending pointerup -> { type: "click" }

Boundary.tsx:266-269
click result -> openExistingEvent(result.event)

Dragging or resizing

Once the pointer moves far enough, or the hold delay finishes, the pending session becomes motion. The overlay appears, the controller writes each frame, and pointer-up returns the moved event only once. 

Controller.ts:318-365
pending -> motion -> scheduleFrame()

Controller.ts:625-699
one RAF loop updates visual + overlay

Dragging while the form is open

If the form was already open for the event, motion closes it while dragging. On release, the adapter restores the moved event into the form instead of immediately submitting. 

Boundary.tsx:247-253
pending -> motion and form was open
-> close form for interaction

Boundary.tsx:62-67
submitMovedEvent restores draft + form

Recurring events

Recurring events do not bypass the existing recurrence decision. After pointer-up, the adapter asks for the update scope instead of saving directly. 

Boundary.tsx:69-74
if recurring:
  requestUpdateScopeForDraft(event)
else:
  submit(event)

Smart scroll

Smart scroll moved into the same RAF loop as timed drag. The controller updates the scroll position and includes that scroll delta in the visual math.

Drag-to-edge week navigation

Edge navigation is also controller-owned. It watches whether the pointer dwells near the left or right edge, applies a week offset to the visual draft, and asks React for one real week change.

The work moved one behavior at a time.

The branch deliberately started with the smallest proof slice: existing timed-event drag, same week, form closed, non-recurring, not pending. Riskier cases stayed on the old path until their turn.

How to understand the code

A beginner reading path

If you are new to this part of the codebase, this order will keep the moving pieces understandable.

Start with the source-of-truth plan

Read controller-v2-plan.md for the rules the branch had to obey. Focus on the ownership rule: React renders saved UI; controller owns live motion; Redux/backend resume after completion.

Then open the boundary

In WeekInteractionBoundary.tsx, read the pointer handlers first. This file answers "how does V2 get control before the old code?"

Read the controller in chunks

In WeekInteractionController.ts, do not read top-to-bottom at first. Read pointer down, pointer up, activation, RAF, teardown, and visual-to-event conversion.

Look at event registration

Read the registration code in GridEvent.tsx and AllDayEvent.tsx. This explains how the controller can find saved events without asking React to move them.

Read the safety guards last

Read motionSafeSelector.ts, useGridLayout.ts, NowLine.tsx, and TimesColumn.tsx. These are not the main feature; they keep unrelated updates from spoiling a clean drag.

Use the benchmark notes as the receipts

Read performance/week-view/README.md from the V2 entries onward. It shows which phases passed, which failed first, which JSON path was used, and why some comparisons needed an explicit saved JSON path.

What remains deliberately legacy The old draft system still exists for things that are not saved event live motion, such as blank-grid create, form editing, and normal draft lifecycle behavior. V2 removed the high-frequency saved-event motion path, not every draft concept in the app.
